# Research Companion — Domain Context

A single-user desktop app (Tauri + Python FastAPI + React) that acts as an AI research and writing companion for books, articles, essays, and investigative journalism.

## Glossary

| Term | Definition | Avoid |
|------|-----------|-------|
| **Buddy** | The user-facing persona name for the AI companion across the entire product. The name the user sees in the UI (chat bubbles, editor callouts, onboarding copy). Internally, different Agent Roles power Buddy at different stages — the user never sees role names. | "assistant", "AI", "agent" (in user-facing copy) |
| **Project** | A user-created workspace scoped to one book or article. Contains a Transcript, Approach, Outline, and Editor tab. Resources are managed globally, not inside the project. Named by the user: Buddy suggests a title at Interview end (which the user accepts or edits), and the title is always inline-editable in the Workspace header thereafter (like a filename rename). Project title is never stored on Resource records — it is resolved at render time from the project ID, so renames propagate everywhere automatically. | "workspace", "session" |
| **Resource** | An uploaded file (PDF, DOCX, TXT) or webpage added to the global library. Does not require a Project — it can exist unattached. Gets vectorized for search and carries Citation Metadata. Identity: SHA-256 content hash for files; normalized URL for webpages. Same-content uploads are silently deduplicated by hash. | "document", "file", "attachment", "material" |
| **Resource Type** | The kind of Resource, chosen by the user at upload time. Determines which Citation Metadata fields are shown. One of: `Book`, `Press/Journal Article`, `Webpage`, `Source Transcript`. Webpages are typed automatically; all other types require explicit user selection. | "format", "category" |
| **Source Transcript** | A Resource Type for the text record of a real-world interview or recording uploaded by the user. Distinct from the Interview Transcript produced by the Project Advisor. | "transcript" (alone, ambiguous) |
| **Citation Metadata** | Structured bibliographic fields attached to a Resource, used to generate references on export. Schema varies by Resource Type: Book (author(s), title, edition, publisher, pub date, ISBN?); Press/Journal Article (author(s), title, journal/outlet, volume, issue, pub date, page range); Webpage (author or organisation, page title, site name, pub date, URL); Source Transcript (attendees, meeting title, date). Auto-extracted on upload, editable by the user. | "metadata", "reference details" |
| **Citation Mark** | An inline TipTap node inserted by the user in the Editor that embeds a Resource ID. Rendered as a superscript marker (e.g. `[1]`). Inserted via slash command (`/cite`) or the Editor toolbar. On export, resolved to a formatted citation and collected into a Bibliography. Removing the parent Resource from the Project orphans the mark, rendering it as `[?]` in the Editor. | "footnote", "reference marker" |
| **Bibliography** | The list of formatted citations appended to an exported document, generated by resolving all Citation Marks in the Editor content against their Resources' Citation Metadata. Formatted in the citation style chosen at export time (APA, MLA, or Chicago). | "references", "works cited" |
| **Export** | The action of producing a DOCX file from the Editor content. Resolves all Citation Marks into a formatted Bibliography appended to the document. User selects citation style (APA, MLA, Chicago) at export time. | "download", "save as" |
| **ResourcesPage** | The full-page management surface for the global resource library. Reached from the sidebar Resources nav button, or from the workspace header Resources chip (which opens it pre-filtered to the active project). Shows all Resources across all Projects, filterable by project. Supports add, delete, and re-index. Clicking a resource row opens the Resource Detail Panel. Distinct from the ResourcesPanel. | "Resources tab", "resources view" |
| **Resource Detail Panel** | A drawer that opens within ResourcesPage when the user clicks a resource row. Shows and allows editing of all Citation Metadata fields for that Resource Type, plus a Projects multi-select for attaching or detaching the Resource from Projects. Closed with Esc or a close button. Read-only fields (content hash, Indexing Status) are shown but not editable. | "edit modal", "resource editor" |
| **ResourcesPanel** | A search-first slide-over that opens over the active view without pushing content. Opened via ⌘K or the sidebar "Search resources" button only — not from the workspace header chip. Read-only — no add or delete. Searching returns both resource-level hits and matching Chunks (semantic + keyword). Backdrop blur is used here and nowhere else in the UI. | "resources drawer", "search panel" |
| **Add Resource Modal** | The modal used to add one or more Resources to the library. Has two layers of segmented controls: (1) File \| URL — determines the entry path; (2) Book \| Press/Journal Article \| Source Transcript — shown only for the File path, determines the Citation Metadata schema. URL path locks the type to Webpage automatically. Metadata fields below the controls adapt to the selected type. Supports batch file uploads; all files in a batch share the same Resource Type (set once for the batch). Each file's Title defaults to its filename and is editable. When two files in a batch produce the same base title, the first 6 characters of the content hash are appended as a collision suffix (e.g. `paper (a3f2c1)`). Author defaults to blank and is editable post-upload. "Attach to project" is optional — Resources can exist in the library unattached to any project. If the content hash already exists in the library, the duplicate is silently deduplicated and the existing Resource is returned instead of re-indexing. | "upload modal", "add modal" |
| **Chunk** | A text segment produced by splitting a Resource during indexing using the Chunker. The unit stored in the Vector Store. | "segment", "passage" |
| **Chunker** | The strategy used to split a Resource's text into Chunks. Two implementations: `RecursiveChunker` (regex-based splits at paragraph → sentence → word boundaries) and the **Semantic Ingester** (LLM-driven boundary detection — see below). Abstract protocol — swappable without changing callers. `chunker_id` is stored per Chunk (a single Resource may have chunks from both paths if Semantic Ingester batches fall back). | "splitter", "text splitter" |
| **Semantic Ingester** | LLM-driven chunker. Pre-splits each extracted page into numbered paragraphs locally, then asks an OpenRouter model to emit `{chunks: [{start: [page, para], end: [page, para]}], skip: [[page, para]]}` JSON identifying chunk boundaries and boilerplate paragraphs. The backend slices the verbatim text using those indices — the LLM never reproduces the document text. Per-batch: strict validation, one retry, then fall back to `RecursiveChunker` for that batch only. `batches_total` and `batches_fallback` are recorded on the Resource so the UI can surface mixed-quality ingests. | "LLM chunker", "semantic chunker" |
| **Embedding** | A dense vector representation of a Chunk, produced by the Embedder and stored in sqlite-vec. | "vector", "encoding" |
| **Embedder** | The strategy used to produce Embeddings from Chunk text. Abstract protocol — swappable without changing callers. Default: `fastembed` running `BAAI/bge-small-en-v1.5` in-process (ONNX, ~65 MB model, ~100 MB RAM, no server needed). Ollama is available as an override for power users. `embedder_id` is stored per Resource so the system knows if re-indexing is needed after a strategy change. | "embedding model", "embedding service" |
| **Vector Store** | The sqlite-vec extension that holds Chunks and their Embeddings. Shared globally across Projects to avoid re-vectorizing the same Resource. | "vector database", "index" |
| **Agent Role** | A named function in the system (Approach Explorer, Research Agent, etc.) that maps to a configured LLM model. | "agent", "task" |
| **Model Router** | The backend module that resolves an Agent Role to an OpenRouter model ID and constructs the LLM client. Reads the user-configured model ID from Settings; falls back to the Catalogue default if none is set. | "model selector", "router" |
| **Catalogue** | The hardcoded table of default OpenRouter model IDs per Agent Role, embedded in the Model Router. Used as defaults when the user has not configured a custom model ID. Displayed read-only in the Settings page as guidance. | "model list", "registry" |
| **Settings** | Non-sensitive configuration (display name, model ID per Agent Role, search provider, Ollama endpoint, embedder override) stored in a JSON file. | "config", "preferences" |
| **Key Store** | The OS credential store (Windows Credential Manager / macOS Keychain / libsecret) accessed via Python `keyring`. | "secrets store", "vault" |
| **Interview** | The AI-driven conversational flow that precedes Project creation. Buddy (powered by the Project Advisor role) asks 3–5 follow-up questions, then signals readiness by asking the user to proceed to the Approach Explorer — that question acts as the natural transition gate. A "Skip to approach →" link appears after the user's first message as an early-exit escape hatch. The Interview is ephemeral — nothing is persisted until it ends. When the Interview ends (Buddy's signal or user skip), the Project is created, the Transcript is saved, and the user enters the Workspace. Runs pre-project in HomeChat. | "onboarding", "wizard", "setup chat" |
| **Transcript** | The saved record of an Interview: the full chat log plus an AI-generated summary of goals and constraints. Stored per Project. Displayed read-only in the Transcript tab. | "chat history", "session log" |
| **Project Advisor** | Agent Role that runs the Interview — grills the user to establish topic, Document Type, and enough context to generate a Transcript summary. | "setup agent", "onboarding agent" |
| **Document Type** | The kind of writing the user is producing (Book, Article, Essay, Investigative Journalism), established during the Interview — never selected via a dropdown. | "genre" |
| **Approach** | The single strategic framing chosen for a Project: how the document will be written and argued. Shape: `{title, description}`. Derived from the Transcript. Replaces the former Angle + Theme split. | "angle", "theme", "purpose", "lens" |
| **Approach Explorer** | Agent Role that generates 3 candidate Approaches from the Transcript. The user picks one, edits inline, or requests a fresh batch. Confirms a single Approach to lock it. | "angle explorer" |
| **Outline Generator** | Agent Role that takes the confirmed Approach and generates a full hierarchical document outline in one step. | |
| **Research Agent** | Agent Role that performs web search + RAG to produce source-backed summary cards. | |
| **Literature Review Synthesizer** | Agent Role that sweeps sources, curates, and synthesises a literature review section. | |
| **Editor AI** | Agent Role that annotates a draft section with style/clarity and factual-grounding suggestions. | |
| **Indexing Status** | The current state of a Resource's vectorization pipeline. One of: `queued`, `indexing`, `ready`, `failed`. A Resource is searchable only when `ready`. Displayed as a badge on the Resource in the Resources tab. | "upload status", "processing state" |
| **Job Tray** | A persistent status bar at the bottom of the UI showing active and recently completed indexing jobs, including progress and any errors. Visible across all tabs while indexing is in progress. | "taskbar", "activity bar", "status bar" |
| **Resource Search** | Semantic search over the Chunks of all Resources in the global library. Accessed via the ResourcesPanel (⌘K). Returns Resource Search Results across all projects. Only searches Resources with Indexing Status `ready`. | |
| **Resource Search Result** | A single hit from a Resource Search: the matched Chunk text, a relevance score, and the Citation Metadata of the parent Resource. | |
| **Source Card** | A UI element summarising one research source: title, excerpt, relevance score, citation. | "result", "card" |
| **Search Provider** | The configured web-search backend (default: Tavily). Configurable in Settings. | "search engine" |
| **Ollama** | Optional local inference server used as an Embedder override for power users who want a different embedding model. Never used for LLM generation. Not required by default — the default Embedder (`fastembed`) runs in-process. | |
| **Sidebar Footer** | The bottom strip of the sidebar showing the user's avatar (initials), display name (configurable in Settings, defaults to OS username), and no plan line. Clicking it opens a dropdown with an "Account" option that navigates to the Account page. | |
| **Account Page** | A placeholder full-page view reached from the Sidebar Footer dropdown. No functionality yet — reserved for future account/subscription features. | |

## Data layout

```
~/.research-companion/
├── settings.json          # Settings (non-sensitive)
├── resources.db           # Global SQLite + sqlite-vec: Resources, Chunks, Embeddings (shared across Projects)
├── sources/               # Raw uploaded files, keyed by content hash (shared across Projects)
└── projects/
    └── <project-id>/
        └── db.sqlite      # Per-project SQLite: project_meta, transcript, approaches, outline_sections, documents, project_resources (join table)

OS Credential Store        # Key Store: OpenRouter API key (and optional provider keys)
```

## External services

| Service | Purpose | Key stored in |
|---------|---------|--------------|
| OpenRouter (`openrouter.ai/api/v1`) | All LLM generation via OpenAI-compatible API | Key Store |
| fastembed (`BAAI/bge-small-en-v1.5`) | Default in-process Embedder — no server, no key needed | — |
| Ollama (`localhost:11434`) | Optional Embedder override for power users — no key needed | — |
| Tavily | Web search for Research Agent | Key Store |
